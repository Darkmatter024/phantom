PHANTOM

What Makes It Special

A field tool built for the data center technician with hands in the rack.

This document explains what PHANTOM is and why it exists differently than every other operations tool out there.

// Thesis

PHANTOM is a single HTML file that runs offline on a phone, holds the cryptographic and human attestation chain for an entire data center build, and never sends a byte of operational data anywhere it doesn't have to.

That sentence is doing a lot of work. The rest of this document unpacks it.
// 01

The Problem PHANTOM Solves

Every operations tool in the data center industry is built for a desk. PHANTOM is built for the rack.

Enterprise DCIM platforms (Sunbird, nlyte, Schneider EcoStruxure) assume a desktop and a stable network connection. CMMS systems (IBM Maximo, ServiceNow) are built for office workflow. ITSM tools optimize for ticket throughput. None of them survive the actual operating environment of a data hall build: gloved hands, dim aisle lighting, one-handed phone use, and intermittent everything (Wi-Fi, Bluetooth, attention, time).

PHANTOM exists because no tool could meet the technician where the work actually happens. So one was built from the ground up around that constraint — with non-negotiable principles drawn from what fails in real environments.

The result is a tool that holds the entire chain of custody for a CoreWeave data hall build on a single phone, with cryptographic source authentication, human attestation gates, and a tamper-evident audit log — all without depending on a network connection.

// 02

The Three Architectural Commitments

Three commitments shape every line of code in the app. Every feature is evaluated against these. Anything that violates them is rejected — even if it would be useful.

Commitment 01

Offline-First, Not Offline-Capable

There is a critical difference between offline-capable and offline-first. Most apps are offline-capable: they degrade gracefully when the network drops. PHANTOM is offline-first: it does not assume a network exists at all. The network is treated as a luxury, not a precondition.

What that means concretely:

  • ▸ The entire application — UI, logic, state, telemetry, audit log — lives on the phone in localStorage. ~1.55 MB of single-file HTML, served once from GitHub Pages, then cached forever by the service worker.
  • ▸ The phone can drop into airplane mode for an entire 12-hour shift, work a complete deployment from EDP intake through phase signoff, and PHANTOM functions identically.
  • ▸ There is no "sync now" button. There is nothing to sync. Local state is canonical state.
  • ▸ AI calls (the only thing that genuinely requires the network) fail gracefully and route to a deterministic local parser when the Cloudflare Worker is unreachable.

This isn't laziness about networking. It's a deliberate choice rooted in field reality: the phone in the cold aisle does not have a reliable network, and any tool that assumes it does will fail in production.

Commitment 02

Single-File, No Build Pipeline

PHANTOM is one HTML file. Approximately 31,200 lines. ~1.55 MB. There is no build step. There is no bundler. There is no npm install. There is no Docker container. There is no deployment pipeline.

Edit the file. Push the file. The file is the application.

Why does this matter?

  • ▸ Auditability: the entire application source is readable in one sitting. There are no transitive dependencies, no minified bundles, no source-maps to chase, no obscured behavior. What you see is what runs on the phone.
  • ▸ Reproducibility: the file at commit SHA abc123 will produce identical behavior on any phone, forever. No "works on my machine" caused by Node version drift or npm resolver luck.
  • ▸ Survivability: the file can be picked up and read like a book. No mystery toolchain. No abandoned dependencies. No webpack.config.js rituals.
  • ▸ Speed: a change can be made, tested, and deployed in under 60 seconds. Edit → push to GitHub → hard refresh on phone → verified live. No CI pipeline waiting 12 minutes to validate that a variable was renamed correctly.

The downside is honest: a 31,200-line single file is unconventional. Modern frontend orthodoxy says to split into modules, components, build artifacts. PHANTOM rejects that orthodoxy because the orthodoxy was built for office software, not for field tools that need to be readable by anyone who picks them up.

Commitment 03

The Cold Aisle Filter

Every UI decision in PHANTOM is evaluated against a single test: would this work for a gloved hand in dim light, one-handed, under physical stress?

This is not a marketing slogan. It's a literal review criterion. Specific consequences:

  • ▸ Tap targets are oversized — minimum 44×44 pt, often 56+ pt for primary actions. No fiddly small buttons.
  • ▸ Critical state is communicated visually first, textually second. Color-coded left rails. Glowing borders on verified states. Hard-stop red on rejection.
  • ▸ No surprise dialogs. No modal interruptions that lose context. Workflows checkpoint state on every meaningful tap so an interruption doesn't lose work.
  • ▸ Display fonts (Orbitron) and tactical numerics (JetBrains Mono) chosen to be legible at small sizes in low light.
  • ▸ The boot screen exists not for branding but to give the phone a calm visual moment before the operational surface appears — protecting against cognitive whiplash on every page load.

Many tools claim to be "mobile-friendly." PHANTOM is physically usable in a cold aisle at 3 AM. That's a measurably different bar.

// 03

The Security Model

PHANTOM holds security-relevant data: deployment manifests, sign-off chains, EDP source documents, build lead identity, audit attestations. Compromising any of those would let a bad actor falsify a CoreWeave data hall build record. The security model is not an afterthought.

3.1

The Two-Gate Pipeline (the May 6, 2026 hardening)

Before May 2026, PHANTOM ingested EDPs blindly. Paste text and the parser would run. Whatever was in the paste became the deployment of record. There was no source-authentication and no human attestation gate.

That changed with the Phase 1A + 1B ship. Today, no EDP becomes a deployment without passing two independent gates:

Gate 1 — Hash Gate (cryptographic source authentication). When an EDP is pasted, PHANTOM does not parse it. It opens the Hash Gate overlay and requires the SHA-256 hash of the EDP that engineering provided through a separate channel (Slack DM, signed email). PHANTOM hashes the pasted EDP on-device using the Web Crypto API (crypto.subtle.digest('SHA-256')) and compares.

If the computed hash matches the engineering-provided hash, PHANTOM writes an EDP_INTAKE_HASH_VERIFIED audit event and continues. If the hashes don't match — the parser never runs. PHANTOM writes an EDP_INTAKE_HASH_MISMATCH audit event with both hashes and the rejection is recorded with full chain-of-custody. The bad EDP cannot become a deployment, no matter how many retries.

This catches: tampered EDPs in the email chain, accidentally-pasted-wrong-EDPs, the wrong revision being pasted, and most adversarial scenarios involving compromise of any single channel between engineering and the field.

The cryptography is real. The hash is computed in the browser via Web Crypto API — no server round-trip, no cloud dependency, no shared secret. The avalanche effect (a single character change produces a completely different hash) is visible in real-time. The implementation is auditable in 30 seconds: it's crypto.subtle.digest('SHA-256', new TextEncoder().encode(text)) — three lines of standard browser API.

Gate 2 — Intake Gate (human attestation manifest). After the Hash Gate clears and the parser runs, PHANTOM does NOT immediately create the deployment. It opens the Intake Gate overlay, showing the parsed counts (racks, fiber connections, optic SKUs, PDUs) and a 6-line verification checklist. Each parsed dimension must be visually confirmed against the EDP source — project metadata, rack manifest, fiber schedule, optic BOM, power schedule, parser warnings — before the deployment is finalized.

Critically: Item 6 (Parser Warnings) is locked until the warnings panel is acknowledged. It is impossible to mindlessly tap through warnings; the warning panel must be physically expanded before the lock releases. This is enforced architecturally, not by trust.

On confirm, PHANTOM writes EDP_INTAKE_CONFIRMED with the EDP hash, parsed counts, warning count, and the build lead's identity. On reject, EDP_INTAKE_REJECTED is written with the partial verification state. Either way, the action is non-repudiable.

3.2

Why Two Gates Instead of One

The temptation is to combine: one screen, hash field plus checklist, single confirm. PHANTOM rejects this. The two gates serve different purposes and combining them dilutes both:

  • The Hash Gate is mathematical truth. Either the hashes match or they don't. There is no human judgment involved. A 7th checkbox saying "I confirm the hash matches" would be meaningless — the math already confirmed it.
  • The Intake Gate is human attestation. The parsed output is being asserted to represent the EDP correctly. This is judgment, not math.

Mixing them creates click-fatigue, where every box gets a mindless "yes." Separating them enforces the discipline that each gate is doing its own job.

3.3

The Audit Log — Tamper-Evident Attestation

Every meaningful action in PHANTOM writes to an append-only audit log keyed by deployment ID. Thirteen distinct event types are tracked: hash verification, intake confirmation, phase signoffs, gate overrides, build lead assumption, rack assignments, scan collections. Each event captures: timestamp, action type, entity affected, summary text, and the actor's identity.

The audit log is tamper-evident, not tamper-resistant. Anyone with physical access to the phone can edit localStorage. PHANTOM is honest about this. What it provides is a chronologically structured append-only ledger where any tampering becomes visible against expected workflow patterns. The choice to be tamper-evident rather than tamper-resistant is deliberate: full tamper-resistance would require server-side log shipping, which would violate the offline-first commitment.

For the operational reality of a CoreWeave data hall — where the threat model is "an honest workflow under time pressure might cut a corner," not "a sophisticated nation-state attacker with physical phone access" — tamper-evident is the correct calibration.

3.4

RBAC — The Build Lead PIN

Some operations require Build Lead privileges: gate overrides, force-mode toggles, sensitive config edits. PHANTOM enforces this through a PIN-based session model:

  • The PIN is set on first run via rbac_promptSetPin. It's hashed using SHA-256 with a per-install random salt — the hash and salt are stored in localStorage, the plaintext PIN never is.
  • Verification uses crypto.subtle for the hash comparison. Constant-time comparison is not provided by JavaScript, but for a 4-digit PIN with a 5-attempt soft lockout and 30-second cooldown, the timing-attack surface is irrelevant.
  • The Build Lead session itself lives in sessionStorage — it never persists across sessions. Closing the browser ends the session.
  • Lost PINs require clearing browser site data. There is no recovery path. This is by design: a recovery mechanism would be the weakest link in the access control.
3.5

XSS Defense

Every user-supplied string rendered to innerHTML in PHANTOM passes through escHtml(). This is enforced consistently across every surface — voice transcriptions, scan-gun output, Ghost Echo entries, OmniNote, audit event summaries, the Intake Gate detail panels. The v3 security audit found two RED findings (apostrophe-in-voice-transcription XSS and conditional XSS in QR-scanner fallback); both were fixed architecturally, not patched. The audit framework now treats any new innerHTML sink as a finding by default.

3.6

What PHANTOM Does Not Defend Against

Honest scoping:

  • EDP source authenticity at the engineering side — PHANTOM verifies the EDP matches a hash provided by engineering. If engineering's own systems are compromised and a bad EDP is signed, PHANTOM will faithfully ingest that bad EDP. Engineering source security is a separate workstream.
  • Server-side audit log integrity — there is no server-side log. Tamper-evident, not tamper-resistant.
  • Sophisticated adversaries with sustained device access — PHANTOM is a field tool, not a hardened security appliance. A nation-state attacker with sustained physical access to an unlocked phone can do things PHANTOM cannot prevent.

These limits are stated explicitly because misrepresenting them would be worse than acknowledging them.

// 04

The Engineering Discipline That Keeps PHANTOM Stable

A 31,200-line single-file application sounds like it should collapse under its own weight. It hasn't. The reason is a set of operating disciplines that are followed without exception.

4.1

The OODA Loop

No global rewrites. No "let's just refactor this whole section." Every change is proposed before execution, scoped to specific surgical edits, and verified against the existing behavior. After the v3 refactor near-disaster — where a function rename silently broke every onclick handler on installed iOS PWAs — the rule became permanent: no function rename touching an HTML onclick handler ships without a service worker cache version bump or it gets rejected.

4.2

The Triple-Check Mandate

Every change passes three independent verifications before handover:

  • phantom_static_analysis.py — a custom analyzer that catches CSS brace imbalance, namespace integrity violations, dead code, and structural anomalies.
  • phantom_smoke.js — a 50-test harness that verifies critical user-facing behaviors against a seed deployment.
  • node --check JS parse validation — catches syntax errors that would silently break the app on iOS Safari (which has different error reporting than Node).

A change that fails any of the three is not shipped.

4.3

The Sniper, Not the General

PHANTOM is for the operator with hands in the rack. It is explicitly not a management dashboard, not a metrics-for-leadership tool, not a compliance-reporting interface. Features that would serve management at the cost of complicating the field workflow are rejected.

This sounds simple but is uncommon. Most operations tools accumulate management-facing features over time. PHANTOM resists this with stated intent. The result is a tool that does its actual job — because it isn't compromised by goals that aren't its own.

4.4

The Cold Aisle Filter as Final Authority

When two design choices conflict — performance vs. clarity, density vs. tap-target size, feature richness vs. cognitive load — the Cold Aisle Filter wins. This is why the app has 56-pt action buttons instead of 32-pt. Why the boot screen has a calm pause instead of a fast splash. Why audit logs render in JetBrains Mono instead of a denser proportional font. Why information that could fit on one screen sometimes doesn't, because the gloved hand needs the breathing room.

// 05

What This Adds Up To

PHANTOM is not a better DCIM tool. It is a different category of thing.

It is a field tool with cryptographic and attestation discipline that holds up under audit.

It is offline-first to the core, not because it's a feature but because the alternative would fail in the actual environment.

It is single-file and auditable because the alternative would be a black box that can't be reviewed.

It is disciplined about scope — every gate, every audit event, every UI decision has a stated reason. Features that don't serve the operational reality are rejected.

It runs reliably on the phone in the technician's pocket regardless of network conditions, battery state, or shift fatigue. And it works.

That is what makes it special.

The Honest Take

⚠ Limits & Trade-offs

PHANTOM is also, in many ways, an unfinished project. Since the May 6 hardening, additional work has shipped: the Manifest tab and revision diff for replaceable EDP records, the Discrepancy Log capture flow for documenting field divergences from the manifest with photo evidence, the Rack Map architectural refactor (search-driven instead of chip strip — scales to 60+ racks at AUS-01), and the Active Deployment scoping pattern — a single source of truth (phantom_active_deployment) that disambiguates which EDP a tech is working in when multiple are open simultaneously, with explicit cross-deployment confirm modals so context switches are never silent. The U-axis was flipped to physical-rack convention (U1 at bottom), the rack header gained deployment context as the first metadata segment, and the EDP Hash Gate gained a 5-tap dev escape hatch for testing without compromising the security model in field use.

What remains is real and substantial: Discrepancy Log list/filter/resolve (Phase 2.5), Delivery Variance Tracker (Phase 3), Phase-to-EDP backreference and Revision Impact (Phase 3), Closeout Manifest generator (Phase 4), Soft Phase Locks and Handoff (Phase 4), Manifest History Toggle and Crash Recovery expansion (Phase 5), and Build Pace Telemetry plus SW hardening (Phase 5, droppable). What ships today is a foundation actively being hardened, not a finished product.

The single-file architecture has trade-offs. The entire file has to be understood before contributing meaningfully. Module isolation is poor. Some sections of the codebase have grown faster than the discipline could refactor them.

The offline-first commitment means certain features are harder than they would be in a server-backed app. Multi-device synchronization is not solved (and Phase 2 QR-code shift bumps will be a labor-intensive partial answer).

The Cold Aisle Filter sometimes makes the app feel sparse compared to feature-dense competitors. That is intentional, but it's worth naming.

These limitations are stated because pretending they don't exist would be worse than acknowledging them. PHANTOM is special because of the choices it makes, and every choice has costs. The costs are accepted in service of the field reality.
