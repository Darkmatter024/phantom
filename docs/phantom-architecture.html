§1 Overview

PHANTOM is a single-file Progressive Web App built for the CoreWeave AUS-01 floor team running rack-and-stack on GB300 NVL72 deployments. It collapses the deployment lead's day-0 toolset — EDP intake, manifest review, rack map, fiber/optic reference, ghost echo capture, voice assistant, shift handoff — into one HTML file that runs offline in iOS Safari.

The Cold Aisle Filter

Every line of code answers one of three questions, or it is bloat and gets cut:

  • Does it speed up validation? — the lead is racing a deployment clock.
  • Does it protect offline data? — there is no signal between the racks.
  • Does it clarify vendor oversight? — handoffs and audits are load-bearing.

Tactical UI doctrine: gloved hand, fast movement, high contrast, large tap targets (44–56 px), monospace tabular data for instant legibility, zero decorative animation that costs Safari render time.

Day 0 Scope

Day 0 is the deployment lead's morning-to-end-of-shift loop:

  • Receive EDP → verify hash → parse to racks/optics/port-maps
  • Walk the floor: scan racks, log discrepancies, capture ghost-echo quirks, raise blockers
  • Generate manifests, run port-map validation, mark phases complete with PIN-gated overrides
  • Hand off the shift with an AI-summarized brief and a printable audit log

Offline-First Commitment

localStorage is the entire persistence layer. There is no remote database. The service worker precaches the HTML, three vendor libraries (ZXing for scanning, SheetJS for XLSX, lz-string for compression), and version.json. AI calls are the only network request and they cleanly degrade — a downed Cloudflare Worker never takes the app down.

No Build Step

One HTML file, one service worker, one JSON sentinel. Every release ships as three files in a zip. No bundler, no transpiler, no framework. iOS Safari runs it as-shipped.

Request Flow

The service worker (sw.js) intercepts only same-origin http(s) requests, scheme-guards everything else (chrome-extension:, data:, blob:, file:), and bypasses cross-origin requests entirely. That bypass is what keeps AI traffic flowing directly to the Cloudflare Worker proxy without ever touching the cache. The Worker holds the Anthropic API key in encrypted env vars, enforces an Origin allowlist, and applies a server-side spend cap.

Why a proxy? No API key reaches the client. A leaked PWA build can't burn billing. Origin enforcement and spend cap both live server-side. The client only knows the proxy URL: https://phantom-api.wfj6t2fk7w.workers.dev/v1/messages.

Cache Eviction Contract

CACHE_VERSION in sw.js and PHANTOM_APP_VERSION in dct-ios.html MUST bump in lockstep with the version field in version.json. On activate, sw.js deletes every cache whose key doesn't match CACHE_VERSION, then calls clients.claim(). The in-app pill (sw_pillRefresh) polls version.json and prompts the operator to tap-to-update when a new build is staged.

Resilient Precache (v1.6.29)

Precache uses Promise.allSettled per URL, not cache.addAll. A single failing precache entry no longer aborts the install. Failures are logged, not fatal.

§3 Navigation Map

Five bottom-nav tabs, in lifecycle order: MASTER = site, BUILD = job, NOW = shift, SCAN = tool, KNOW = reference. The bottom-nav comment in source calls this the "4-Verb Collapse" — visible verbs are NOW · BUILD · SCAN · KNOW, with MASTER added in v1.6.30 as a fifth slot to host the site reference.

Page IDs vs. nav labels The visible nav labels were collapsed to the 4-Verb vocabulary, but the underlying showPage() still routes by legacy page IDs: pg-sop (BUILD), pg-fiber (KNOW), pg-scan (SCAN). There is no pg-now — NOW resolves through showPage('triage') into today_render(). There is no pg-build or pg-know.
